# Backlog / 将来対応メモ

MVP には含めていない、将来対応したい項目の記録。

## 権限管理: 削除は作成者または上位権限ユーザーのみ

**やりたいこと**
- アイデアの削除（`delete_idea`）は、**そのアイデアを作成したユーザー本人**、
  もしくは**上位権限（管理者など）を持つユーザー**だけが実行できるようにする。
- それ以外のユーザーからの削除はサーバー側で拒否する。

**なぜ MVP では未対応か**
- 現在は認証・ユーザー管理が無く、「誰がリクエストしているか」をサーバーが識別できない。
- そのため「作成者かどうか」「権限があるか」を判定する材料が存在しない。

**実装に必要な前提（おおよその段取り）**
1. **呼び出し元の識別** — API キー認証（Bearer トークン）や Cloudflare Access で
   リクエストを行ったユーザーを特定できるようにする。
2. **作成者の記録** — `ideas` テーブルに `author_id`（作成者）カラムを追加し、
   `create_idea` 時に呼び出し元ユーザーを保存する。
3. **ユーザーと権限の管理** — `users` テーブル等にロール（例: `member` / `admin`）を持たせる。
4. **削除時の権限チェック** — `delete_idea` で
   「呼び出し元 == 作成者」または「呼び出し元が管理者ロール」を満たす場合のみ
   物理削除を許可し、そうでなければエラーを返す。
   - 同様の制御を `update_idea` / `advance_status` にも広げるか要検討。

**備考**
- これは下記「セキュリティ」項目（認証・ユーザー管理）と密接に関係する。
  認証基盤を入れる際にまとめて設計するのが望ましい。

## 関連する将来拡張（参考）

### 機能拡張
- `users` テーブル + `author_id` / `assignee_id` → 作成者・担当者管理
- `comments` テーブル（種別: コメント / ステータス変更）→ 活動履歴・監査ログ
- `idea_tags` テーブル（多対多）→ タグ絞り込み
- アイデア全文検索（FTS5 もしくは LIKE 検索）

### セキュリティ（本番DB移行前に必須）
- API キー認証（Bearer トークン）→ Cloudflare Secret に保存
- Cloudflare Access（Zero Trust）→ 会社メールドメインのみ許可、SSO 連携
- 操作ログ（誰がいつ何を変更したか）→ インシデント追跡
- Row Level Security →「自分担当のみ見える」等を DB 層で強制
- DB を Supabase（PostgreSQL / Tokyo リージョン）へ移行 → 日本リージョン固定・RLS 対応
